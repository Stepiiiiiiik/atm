# ATM (Банкомат)

Данный проект представляет собой реализацию системы банкомата в рамках лабораторной работы №5. Основная цель — освоение многослойных архитектур и паттернов проектирования, в частности **гексагональной архитектуры** с **анемичной моделью домена**.

## Цель работы
Проверить освоение студентом принципов построения многослойных приложений, разделения ответственности, внедрения зависимостей и организации кода с использованием современных подходов .NET.

## Функциональные возможности
- Создание пользовательской и административной сессий.
- Создание банковского счёта.
- Просмотр баланса счёта.
- Снятие денег со счёта.
- Пополнение счёта.
- Просмотр истории операций по счёту.
- Любая операция со счётом автоматически фиксируется в истории.

## Архитектура и технологии
Проект построен в соответствии с принципами **гексагональной архитектуры** (порты и адаптеры) с использованием **анемичной модели домена**: сущности домена не содержат бизнес-логики, а выступают в роли простых контейнеров данных; вся логика вынесена в сервисы приложения.

### Используемые технологии и библиотеки
- **.NET 6/8** (зависит от настройки проекта)
- **ASP.NET Core** — для создания Web API (контроллеры, не Minimal API)
- **Microsoft.Extensions.DependencyInjection (MSDI)** — для связывания абстракций и реализаций (каждый модуль предоставляет свой метод расширения для регистрации сервисов)
- **NSubstitute** — для создания мок-объектов в тестах
- **PostgreSQL** — реляционная база данных для хранения информации о счетах, операциях и сессиях

### Ключевые архитектурные решения
- **Сессии**: при создании сессии (пользовательской или административной) генерируется уникальный ключ (`Guid`), который используется для авторизации последующих запросов. Данные сессий хранятся в базе данных через репозитории.
- **Инверсия зависимостей**: все зависимости направлены внутрь, внешние слои зависят от абстракций, определённых во внутренних слоях.
- **Тестируемость**: бизнес-логика покрыта модульными тестами с использованием моков репозиториев, что позволяет тестировать сценарии без реального хранилища.
- **Анемичная модель домена**: сущности не содержат методов, вся логика реализована в сервисах приложения.
